# Phase 7: AWS Deployment (Terraform IaC)

We have successfully architected and implemented the Infrastructure as Code (IaC) required to deploy ArborWatch to a production AWS environment. This setup utilizes serverless Fargate containers, managed databases, and scalable object storage.

## AWS Infrastructure Modules Created

### 1. Networking (`vpc.tf` & `security_groups.tf`)
- **VPC** deployed with CIDR `10.0.0.0/16`.
- **Public Subnets**: Route to the Internet Gateway, housing the Application Load Balancer and NAT Gateway.
- **Private Subnets**: No direct internet access. Houses the ECS Tasks, RDS, and ElastiCache. Outbound internet is routed through the NAT Gateway.
- **Security Groups**: Granular network isolation ensuring the database and cache can only be reached by the ECS Tasks, and the Tasks can only be reached by the Load Balancer.

### 2. Databases & Caching (`rds.tf` & `elasticache.tf`)
- **PostgreSQL / PostGIS (RDS)**: An Amazon RDS instance running PostgreSQL version 16 (matching your local environment). It automatically handles backups and maintenance.
- **Redis (ElastiCache)**: A managed Redis 7 instance that will act as your Celery Message Broker and Django Cache backend.

### 3. Compute & Orchestration (`ecs.tf` & `iam.tf`)
- **ECS Fargate Cluster**: Serverless container hosting, meaning you don't manage EC2 instances.
- **Task Definitions**:
  - `web`: Runs the Gunicorn WSGI server.
  - `celery-worker`: Runs the Celery worker for ML and Ingestion.
  - `celery-beat`: Runs the Celery periodic task scheduler.
- **IAM Roles**: Explicitly granting the ECS Tasks permissions to access the S3 media bucket and push CloudWatch logs.

### 4. Routing & Storage (`route53.tf`, `alb.tf`, `s3.tf`)
- **Route53 & ACM**: Automatically requests a free AWS SSL certificate for `arborwatch.net`, validates it via DNS records, and attaches it to the Application Load Balancer.
- **ALB**: Routes internet traffic securely (HTTPS) to the private `web` Fargate tasks.
- **S3 Bucket**: A dedicated bucket for serving Django Static files and Media uploads using `django-storages`.

### 5. Remote State (`bootstrap/main.tf`)
- I've included a bootstrap module (`terraform/bootstrap/main.tf`). This will spin up an S3 bucket and DynamoDB table that Terraform uses to securely store its state and prevent concurrent modification issues.

### 6. GitHub Actions (`.github/workflows/deploy.yml`)
- We've added an automated CI/CD pipeline! When you push code to `main`, GitHub Actions will authenticate with AWS, build your Docker image, push it to your private ECR, and command the ECS cluster to instantly deploy the new containers with zero downtime.

## What's Next? (How to Deploy Updates)

When you are ready to physically deploy this to AWS, you will need to:
1. Initialize the AWS account and create an IAM Admin User.
2. Run `terraform apply` in the `terraform/bootstrap/` directory to create the state bucket.
3. Uncomment the `backend "s3"` block in `terraform/providers.tf` and fill in the bucket name.
4. Run `terraform apply` in the `terraform/` directory to spin up the world!
5. Add your `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` to your GitHub Repository Secrets to enable the automated CI/CD pipeline.
